# Issue Triage & Fix — 確認→修正→検証→完了ワークフロー

## トリガーワード
「イシュー消化」「バグ修正一括」「issue triage」「まとめて修正」「簡単なものから」

## ワークフロー

### Phase 0: 重複整理
1. `gh issue list --state open` で全イシューを取得
2. タイトル・本文が重複するイシューを特定
3. 最も詳細なイシューを canonical として残し、他を close（コメントで canonical を参照）
4. `gh issue close <num> -c "Duplicate of #<canonical>"`

### Phase 1: 難易度トリアージ
イシューを3段階に分類:
- **S (Simple)**: 1ファイル、10行以内の修正。UI表示バグ、通知ロジック反転など
- **M (Medium)**: 2-3ファイル、機能追加。セッション名統一、モード切替など
- **L (Large)**: アーキテクチャ変更、複数コンポーネント連携。SSE再接続、UI大改修など

**S → M → L の順に着手する。**

### Phase 2: 事象確認（各イシューごとに実行）
1. **コード読解**: 該当箇所のコードを読み、イシューの記述と一致するか確認
2. **再現確認**: 可能ならブラウザ（Playwright MCP）やテストで事象を再現
3. **判定**:
   - 再現した → Phase 3 へ
   - 既に修正済み → `gh issue close` + コメント「既に修正済み」
   - 再現不可 → コメントを残してスキップ

### Phase 3: 修正
1. 最小限の変更で修正（over-engineering 禁止）
2. 関連するイシュー番号をコミットメッセージに含める
3. 型エラー・lint エラーがないことを確認: `npm run build`

### Phase 4: 検証
1. `npm run build` 成功を確認
2. 可能ならブラウザで動作確認（Playwright MCP）
3. 既存テストが通ることを確認: `npm run test`

### Phase 5: 完了
1. 検証OKなら `gh issue close <num> -c "Fixed in <commit-hash>"`
2. TodoWrite で完了マーク
3. 次のイシューへ

## 原則
- **1イシュー1コミット**: 混ぜない
- **確認できないものは飛ばす**: 推測で修正しない
- **既に直っていたら即close**: 無駄な作業をしない
- **破壊的変更は確認を取る**: ユーザーに聞く
