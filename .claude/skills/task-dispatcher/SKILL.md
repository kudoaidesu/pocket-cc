# タスクディスパッチャー

## 原則

窓口は作業しない。最適なワーカーを選び、的確な指示で割り振る。

## ディスパッチフロー

### 1. タスク分析（10秒以内）

受けたタスクを以下で分類：

| 判断 | 基準 | アクション |
|------|------|-----------|
| 軽微 | 1ファイル数行、Git操作、質問回答 | 窓口が直接対応 |
| 単一タスク | 1機能、1Issue | ワーカー1つ |
| 複数タスク | 独立した複数Issue/機能 | ワーカー並列起動 |
| 依存タスク | A完了後にB | ワーカー直列（Aの完了を待ってB起動） |

### 2. ワーカー選択

| ワーカータイプ | 用途 |
|---------------|------|
| `worker-pocket-cc` | pocket-cc のコード変更・修正・機能追加 |
| `worker-eisamap-2025` | eisamap_2025 のコード変更 |
| `Explore` (haiku/sonnet) | 調査・情報収集のみ（コード変更なし） |
| `Plan` | 設計・計画立案のみ |

### 3. 指示テンプレート

ワーカーへの指示は以下を必ず含める：

```
## 対象
- Issue番号 or タスク概要

## 目的
- 何を達成するか（1-2文）

## スコープ
- 対象ファイル/機能の範囲
- やること/やらないことの境界

## 手順
1. 現状確認（コード読解）
2. ブランチ作成（develop から feature/* or fix/*）
3. 実装
4. テスト確認（npm run test）
5. ビルド確認（npm run build）
6. コミット（contextual-commit 形式）
7. PR作成（develop ベース）

## 制約
- any型禁止、ESM使用
- 既存テストを壊さない
- （タスク固有の制約を追記）
```

### 4. 並列起動の判断

**並列OK:**
- 異なるファイルを触るタスク同士
- 異なるプロジェクトのタスク
- 調査タスクと実装タスク

**並列NG（直列にする）:**
- 同じファイルを触る可能性があるタスク
- 片方の結果が他方に影響するタスク
- DB スキーマ変更を伴うタスク同士

### 5. 報告

割り振り直後：
```
[Issue名] → worker-N に割り振りました。
```

完了時：
```
[Issue名] — 完了。PR #xxx を作成しました。
変更概要: （1-2文）
```

失敗時：
```
[Issue名] — ワーカーがエラーで停止。
原因: （要約）
対応: （リトライ / 別アプローチ / ユーザー判断待ち）
```

## ワーカーの品質チェック

ワーカー完了後、報告前に最低限確認：
- PR が作成されているか
- テストが通っているか（ワーカーのログ確認）
- コミットメッセージが適切か
